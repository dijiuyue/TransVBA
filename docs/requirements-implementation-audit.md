# 需求实现审计与修复计划

审计对象：

- 需求文档：`C:/Users/YHR/Desktop/VBA文件/多模板自动格式化与检查插件需求说明书/多模板自动格式化与检查插件需求说明书/多模板自动格式化与检查插件需求说明书.docx`
- 需求摘录：`docs/_requirements_extracted.txt`

结论先说清楚：这个项目不是完全没有实现格式化，而是存在大量“代码看起来写了，但真实文档里不稳定或达不到需求”的断层。最典型的是段前段后：底层函数确实写入了 `w:beforeLines` / `w:afterLines`，但正文、题注、表格单元格等路径没有统一清理旧的 `w:before` / `w:after`，导致已有 Word 文档上的旧格式可能继续影响显示效果。再加上校验器没有覆盖段前段后，用户会看到“点了应用，但格式没变，也没有报错”。

## 现场澄清后的最新口径

以下三点优先级高于需求文档中的旧描述，也高于当前模板 JSON 里的默认值：

1. 所有已纳入模板管理的标题和正文的段前、段后都应该是 `0`。包括一级到四级标题、正文、图题、表题、目录标题、附录标题、附录正文等普通段落格式化入口。不得再保留通用模板一级/二级标题 `0.5` 行的旧配置。
2. 图标题和表格标题必须居中对齐。现在实际表现成“两端对齐”，说明配置值虽然写了“居中”，但应用链路或旧段落格式清理没有真正生效。
3. 表格宽度和行高应该自适应。不能把表格宽度、列宽、行高写死，也不应该用固定 `0.6cm` 行高作为目标。后续表格相关修复应以“删除/避免固定尺寸、启用 Word 自动适应”为准。

## P0 必修问题

### P0-1 段前段后不可靠，且目标应统一为 0

需求：

- 所有标题和正文段落：段前 0 行、段后 0 行。
- 这里的“所有”包括一级到四级标题、正文、表题、图题、目录标题、附录标题、附录正文等。
- 通用模板中一级、二级标题原来配置的 `0.5` 行不再符合现场口径，应改为 `0`。
- 用户实际反馈：段前段后所有都不生效。

当前实现：

- `tvba_core_oox.py::set_before_after_lines()` 写入 `w:beforeLines` 和 `w:afterLines`。
- `tvba_core_title.py::apply_title_style()` 会先 `clear_paragraph_formatting()`，所以标题在干净测试里通常有效。
- `tvba_core_body.py::apply_paragraph()` 不清理原段落格式，只追加或覆盖部分 XML。
- `tvba_core_table.py`、`tvba_core_figure.py`、`tvba_core_toc.py` 等模块也没有统一清理旧 spacing 属性。
- `tvba_core_validate.py` 没有检查“三级标题段前段后是否为 0”的需求。

为什么达不到：

- Word 段落间距可能同时存在点值属性 `w:before` / `w:after` 和行值属性 `w:beforeLines` / `w:afterLines`。
- 当前 helper 只设置 `beforeLines/afterLines`，没有删除旧的 `before/after`。
- 对已有文档，旧样式、旧直接格式、自动段前段后可能残留。用户肉眼看到的结果就会像设置完全没生效。
- 测试只覆盖了“新建干净段落”，没有覆盖“已有脏格式段落”。

修复计划：

1. 在 `tvba_core_oox.py` 增加一个统一 spacing helper，例如 `apply_paragraph_spacing(paragraph_format, before_lines, after_lines, line_spacing=None, line_rule="auto")`。
2. helper 内部必须先标准化 spacing：
   - 删除 `w:before`
   - 删除 `w:after`
   - 删除旧的 `w:beforeLines`
   - 删除旧的 `w:afterLines`
   - 删除旧的 `w:beforeAutospacing`
   - 删除旧的 `w:afterAutospacing`
   - 再写入模板需要的 `beforeLines/afterLines`
   - 明确写入 `beforeAutospacing="0"` 和 `afterAutospacing="0"`
3. 为了兼容 Word 显示，建议当 before/after 为 0 时同时写 `w:before="0"` 和 `w:after="0"`，避免旧样式回灌。
4. 把正文、标题、表题、图题、目录、附录、页眉相关代码都切到这个统一 helper，不要每个模块自己拼 spacing XML。
5. `tvba_core_body.py::apply_paragraph()` 不一定要清空整段格式，但必须清理 spacing、indent、alignment 这类和模板强相关的段落属性。
6. `tvba_core_validate.py` 增加段前段后校验：
   - 所有标题段前段后必须 0。
   - 正文段前段后必须 0。
   - 表题、图题、目录标题、附录标题、附录正文也必须 0。
7. 更新 `templates/general_spec.json` 和 `templates/dapeng_internal.json`：
   - 所有 title level 的 `before_lines` / `after_lines` 改为 `0`。
   - body、table caption、figure caption、toc、appendix 的 before/after 也统一为 `0`。

必须补的测试：

- `tests/test_oox.py`：给段落预先写入 `w:before="480"`、`w:after="480"`，调用 spacing helper 后断言旧属性被删除或置 0。
- `tests/test_body.py`：正文段落带旧段前段后，应用后必须变成模板值。
- `tests/test_title.py`：一级到四级标题带旧段前段后，应用后必须全部 0/0。
- `tests/test_table.py` / `tests/test_figure.py`：表题、图题带旧 spacing，应用后必须 0/0。
- `tests/test_toc.py` / `tests/test_appendix.py`：目录和附录相关段落应用后必须 0/0。
- `tests/test_validate.py`：任何被模板管理的标题、正文、题注段前段后不为 0 时必须报 issue。

验收标准：

- 用一个带大量旧格式的真实 docx 应用模板后，抽查 XML 中目标段落没有残留非 0 的 `w:before/w:after`。
- GUI “格式检查”能明确报出段前段后不符合，而不是静默通过。

### P0-2 四级标题识别规则应改为点分编号，且不要处理五级标题

需求：

- 四级标题应识别类似 `1.1.1.1` 的点分编号格式。
- 不需要处理 `1)`、`1）`、`(1)` 这类括号编号。
- 不需要处理 `a.` 等五级标题。
- 匹配四级标题后需要清除原格式并注入模板四级标题格式。

当前实现：

- `tvba_core_title.py::identify_list_item()` 存在，但它识别的是 `1)` / `(1)` 以及 `a.` 这类列表项。
- 但 `auto_detect_and_format()` 当前主流程主要处理数字标题、中文标题和部分 Word 自动编号标题。
- 现在的问题不是要把 `identify_list_item()` 接入主流程，而是要避免继续沿用这个旧方向。

为什么达不到：

- 旧计划把四级标题理解成 `1)` / `(1)`，这和现场口径不一致。
- 如果 Claude 按旧计划接入 `identify_list_item()`，会引入不需要的括号编号和五级标题修改，反而扩大误判范围。

修复计划：

1. 不要接入 `identify_list_item()` 作为四级/五级标题识别入口。
2. 新增或调整标题识别规则：
   - 一级：按现有规则。
   - 二级：`1.1`
   - 三级：`1.1.1`
   - 四级：`1.1.1.1`
3. 四级标题只匹配点分编号，建议正则类似：`^\d+\.\d+\.\d+\.\d+(?:\s+|　+|$)`。
4. 不新增五级标题处理，不修改 `a.` 相关行为。
5. 对 `1)`、`1）`、`(1)` 明确加保护测试，确保不会被当作四级标题格式化。
6. 注意不要把正文中的普通版本号、日期、编号误判为标题，需要结合以下条件降低误判：
   - 段落长度阈值。
   - 是否以点分编号开头。
   - 是否不是表格内容。
   - 是否不是目录行。
   - 可选：和上下文标题层级关系结合。
7. 识别后只调用 `apply_title_style(para, settings, level=4)`。

必须补的测试：

- `1.1.1.1 适用范围` 应被格式为四级标题。
- `1.1.1.1　适用范围` 应被格式为四级标题。
- `1) 适用范围` 不应被当作四级标题。
- `1）适用范围` 不应被当作四级标题。
- `(1) 适用范围` 不应被当作四级标题。
- `a. 说明` 不应被新增处理为五级标题。
- 正文中类似 `2026.05.16`、版本号、目录行不能误判。

### P0-4 图标题和表格标题居中没有真正生效

需求：

- 图标题和表格标题必须居中对齐。
- 不允许应用后仍显示为“两端对齐”。

当前实现：

- `templates/general_spec.json` 和 `templates/dapeng_internal.json` 里 table / figure 的 `title_alignment` 都是 `居中`。
- `tvba_core_table.py::apply_table_caption()` 和 `tvba_core_figure.py::apply_figure_caption()` 也有 `_ALIGN_MAP = {"居中": 1}`。
- 但真实 GUI 中图题、表题显示成“两端对齐”，说明配置值没有可靠落到最终 Word 段落。

为什么达不到：

- 可能原因之一是旧段落格式或样式的 alignment 残留，直接设置 `para.alignment` 后仍被样式或后续流程覆盖。
- 也可能是题注没有被识别为 caption，导致没有进入 `apply_table_caption()` / `apply_figure_caption()`。
- 还可能是后续正文格式化又把题注当正文处理，覆盖成两端对齐。

修复计划：

1. 先写回归测试复现：表题、图题初始为两端对齐，应用后必须变为居中。
2. 检查 `apply_settings_to_document()` 的顺序，确保题注被格式化后不会再被正文格式覆盖。
3. 在 caption formatter 中清理旧 `w:jc` 后明确写入 `w:jc w:val="center"`，不要只依赖 python-docx 的 `para.alignment`。
4. 如果题注识别失败，扩展识别规则，覆盖：
   - 表题类别：`表` + 可选空格 + 数字/多段点分编号 + 可选 `-序号` + 空白 + 标题文本。
   - 图题类别：`图` + 可选空格 + 数字/多段点分编号 + 可选 `-序号` + 空白 + 标题文本。
   - 示例包括但不限于 `表1-1  测试表格`、`表 1.8.4-2  本工程珠海站场和阀室设置`、`表1.8.12-2  本工程车辆配置一览表`。
   - 示例包括但不限于 `图1-1  测试图片`、`图 1.8.3-1  工艺系统示意图`、`图1.10.1-1  双水首站与已建工程界面划分示意图`。
5. 校验器增加图题/表题对齐检查，非居中必须报 issue。
6. 图题/表题默认特殊缩进必须是“无”。即使原段落已有首行缩进、悬挂缩进、大纲级别，也必须在应用题注格式时清理：
   - 删除 `w:firstLine`。
   - 删除 `w:hanging`。
   - 删除 `w:outlineLvl`。
   - 明确写入居中对齐 `w:jc w:val="center"`。
7. 如果类别测试能通过，但真实 GUI 仍失败，优先排查现场差异：
   - 题注是否在文本框/形状中，而不是普通 Word 段落中。
   - 编号和标题之间是否是不可见字符、域代码、换行符，或被拆成复杂 run。
   - 当前 GUI 是否仍在运行旧进程或旧构建。
   - 完整应用流程是否先格式化题注，后续又被其他模块覆盖。

- `tests/test_table.py`：表题编号类别必须被识别并居中，而不是只测某一个固定标题。
- `tests/test_table.py`：已有首行缩进、大纲级别、两端对齐的表题，应用后必须变为无特殊缩进、无大纲级别、居中。

### P0-5 表格宽度和行高不应写死，必须自适应

需求：

- 表格宽度自适应。
- 表格行高自适应。
- 不应强制固定行高、固定列宽或固定表宽。

当前实现：

- `tvba_core_table.py::apply_table_body()` 调用 `set_table_layout(table, settings.auto_fit_mode)`。
- 同时仍调用 `set_table_borders(table, line_width_pt=settings.line_width_pt)` 和 `set_row_height_at_least(row, settings.row_height_cm)`。
- `templates/general_spec.json` 仍有 `row_height_cm: 0.6`。
- `templates/dapeng_internal.json` 仍有 `row_height_cm: 0.7`。
- GUI 里仍有表格行高、线宽等可编辑项。
- 注意：这里的“表格宽度自适应”指表格整体宽度、列宽、单元格宽度，不是边框线宽。边框线宽可以保留为独立格式项，但不能被误当成表格宽度。

为什么达不到：

- 只要写入 `w:trHeight`，Word 就不再是完全自然自适应行高。
- 如果写入固定表宽、固定列宽或关闭 autofit，表格宽度也不会按内容或页面自动适配。
- 当前需求口径已经明确“自适应”，所以旧计划中“修成 exact 0.6cm”是错误方向。

修复计划：

1. 停止在应用阶段调用 `set_row_height_at_least()`，除非用户显式打开固定/最小行高模式。
2. 表格应用时删除已有 `w:trHeight`，让行高回到 Word 自适应。
3. 表格应用时启用 autofit：
   - `table.allow_autofit = True`
   - OOXML 中使用自动布局，不写固定 `tblW`。
   - 清理会锁定列宽的 `gridCol/@w` 或单元格固定宽度时要谨慎，先通过小样本验证。
4. 模板设置应调整：
   - 移除或忽略 `row_height_cm` 作为默认强制项。
   - 增加明确字段，例如 `auto_fit_rows: true`、`auto_fit_width: true`。
5. GUI 应隐藏或弱化“表格行高/表格宽度”输入，或只有在用户关闭自适应时才启用。边框线宽如仍有需求，应改名为“边框线宽”，避免和表格宽度混淆。
6. 校验器从“检查行高等于某值”改为“检查没有固定行高/宽度锁定”。

必须补的测试：

- `tests/test_table.py`：已有 `w:trHeight` 的表格应用后被移除。
- `tests/test_table.py`：应用后 `table.allow_autofit` 为 True。
- `tests/test_validate.py`：存在固定行高时，如果模板要求自适应，应报 issue。
- `tests/test_validate.py`：不存在固定行高时通过。

## P1 重要问题

### P1-1 配置和最新现场口径不一致

发现：

- `templates/general_spec.json` 中一级、二级标题段前段后仍是 `0.5`。
- 现场已经明确：所有标题和正文段前段后都必须为 `0`。
- 表格的 `row_height_cm` 仍然存在，但现场已经明确表格行高应自适应，不应写死。

修复计划：

1. 直接更新两个模板 JSON，使所有段落类 before/after 默认值为 `0`。
2. 将表格尺寸相关配置从“默认强制值”改成“仅在用户关闭自适应时才使用”。
3. 在 `docs/capability-matrix.md` 或新建验收矩阵里标明“现场澄清覆盖旧需求描述”。

### P1-2 正文格式没有彻底覆盖旧直接格式

需求：

- 一键应用模板后，正文应是宋体/Times New Roman、小四、常规、1.5 倍行距等。

当前实现：

- `apply_paragraph()` 遍历已有 runs 设置字体字号。
- 但不会清理加粗、斜体、下划线、颜色、字符间距等旧直接格式。
- 如果段落没有 run 或存在复杂域、超链接、修订等结构，覆盖不一定完整。

为什么达不到：

- 需求是“清除原格式后注入模板”，当前正文路径更像“局部覆盖”。

修复计划：

1. 为正文提供“强制模板化”模式：
   - 清理 run 的 bold/italic/underline/color/highlight 等常见直接格式。
   - 保留超链接、域代码、图片等结构。
2. GUI 的“修改模式”需要明确文案：只是允许编辑模板参数，不是是否强制清格式。
3. 增加“是否清除原直接格式”的设置，默认按需求开启。

### P1-3 禁忌词只检查，不替换或提示式修正

需求：

- 大鹏模板：不允许出现“附图/附表”，提示改成“附件”。
- 需求特别说明“禁忌词替换”。

当前实现：

- `tvba_core_validate.py` 可以检查 forbidden words。
- 应用阶段没有稳定的模板驱动替换。

为什么达不到：

- 用户点“应用”不会改，点“检查”只会报 issue。
- 如果需求是自动替换或弹窗确认，目前没有实现。

修复计划：

1. 明确产品行为：
   - 只检查并提示。
   - 自动替换。
   - 弹窗列出候选替换，让用户确认。
2. 模板增加 replacement mapping，例如：
   - `"forbidden_replacements": {"附图": "附件", "附表": "附件"}`
3. 应用阶段按 mapping 替换，检查阶段继续兜底。

### P1-4 表题和图题位置没有自动修正

需求：

- 表题在表上方。
- 图题在图下方。

当前实现：

- 代码能识别并格式化部分表题、图题。
- 校验器有位置检查倾向，但应用阶段没有可靠移动段落位置。

为什么达不到：

- 如果原文图题在图上方或表题在表下方，应用格式不会真正把它移到正确位置。

修复计划：

1. 先把“检查”和“自动移动”拆开。
2. P1 阶段至少做到检查可靠，明确提示哪个题注位置错误。
3. P2 再实现自动移动，因为移动 Word XML 段落和图片锚点风险较高。

### P1-5 目录标题下方空行没有实现

需求：

- 大鹏模板目录标题二号、加粗、1.5 倍行距、居中，目录下方空一行。

当前实现：

- `tvba_core_toc.py` 可以识别目录标题并设置字体字号等。
- 没有稳定插入或确保“目录标题下方空一行”。

修复计划：

1. `refresh_toc()` 处理目录标题后，检查下一段是否为空段。
2. 如果没有，插入一个空段并设置合适行距。
3. 如果已有多个空段，按需求压缩为一个。

### P1-6 附录和页眉能力不完整暴露在 UI

需求：

- 大鹏模板要求附录标题、附录正文、页眉 Rev. 后一个空格等。

当前实现：

- 代码有 appendix/header 相关设置和校验。
- GUI 主要暴露正文、标题、表格、图题等，缺少完整的附录、页眉、目录参数编辑入口。

为什么达不到：

- 如果用户通过 GUI 修改模板参数，部分需求字段不可见、不可控。

修复计划：

1. GUI 增加“高级/附录/页眉/目录”设置区域。
2. 如果暂不支持编辑，也要显示当前模板摘要，避免用户误以为所有参数都已可改。

### P1-7 自动编号标题在大文档中存在准确性和性能取舍

背景：

- 最近为了避免 GUI 卡在标题扫描，已经增加了大量自动编号候选时跳过 COM resolver 的保护。

风险：

- 跳过 COM 后，Word 自动编号标题可能无法全部准确识别。
- 这会影响需求里的“自动识别编号结构”。

修复计划：

1. 保留性能保护，但在 GUI 状态或日志中明确提示“已跳过 Word 自动编号深度解析”。
2. 增加一个可选“深度识别模式”，用户愿意等待时再启用 COM。
3. 长期方案是解析 `numbering.xml`，不要依赖逐段 COM 调用。

## P2 完善问题

### P2-1 校验器覆盖不足

需求表里明确要检查：

- 中文非宋体。
- 英文/数字非 Times New Roman。
- 任意标题、正文、题注段前段后不为 0。
- 半角括号、项目符号未带句点。
- 表格字号。
- 表格存在固定行高/固定宽度。
- 封面标题字号。
- 附录正文字号。
- 一级标题网格对齐。
- 图题/表题编号和名称之间空格。
- Rev. 后空格。
- 附图/附表。
- 附件标题缺冒号。

当前状态：

- 已实现一部分。
- 段前段后校验缺失。
- 表格固定尺寸校验缺失。
- 主席令编号校验错位。
- 封面标题和目录空行的检查需要加强。

修复计划：

1. 把校验项按需求表建成 checklist。
2. 每个 checklist 项必须对应至少一个单元测试。
3. GUI 检查结果要显示可定位信息：页/段落/表格/单元格。

### P2-2 Golden 文档回归不足

需求文档提到要参考附件作为视觉验收。

当前状态：

- 已有 `tests/fixtures/golden_compare.py`，但还没有形成覆盖真实需求样例的稳定流程。

修复计划：

1. 为两个模板各准备一个小型输入 docx 和一个 golden docx。
2. 自动化比较：
   - 段落字体。
   - 字号。
   - 粗体。
   - 行距。
   - 段前段后。
   - 缩进。
   - 表格是否存在固定行高/宽度。
3. 对无法稳定比较的视觉项，生成检查报告，人工验收。
